Great question — and yes, your intuition is correct.

Short answer

In a pure web app running in the browser (this project), you cannot silently write to an arbitrary install folder on phone/computer like a desktop app would. The browser sandbox blocks that for security reasons.
Right now the app uses localStorage intentionally.

Why this app currently behaves that way

  • Persistence is explicitly implemented with localStorage.getItem/setItem.

  • App state is auto-saved on each relevant state change via saveAppState(...).

  • The README also documents browser local storage persistence.

Can we save “on device” but not localStorage?

Yes, but with constraints:

  1. Manual file export/import (already exists)

    • User downloads JSON to device storage and can re-import later.

    • Most portable and reliable across browsers.

  2. File System Access API (web)

    • Possible on some platforms/browsers (best on desktop Chromium).

    • Requires user permission and file handle flow.

    • Not broadly consistent on iOS/Android browsers.

  3. Turn app into native wrapper (Capacitor / React Native / Flutter wrapper)

    • Then you can write to app-specific device storage directly.

    • Best for phone-first “installed app” behavior.

So: for the current architecture (Vite + React browser app), localStorage + backup export is the normal safe pattern. If you want “real file persistence without manual export,” we’d need either File System Access integration (limited mobile support) or a native wrapper path.

What “web fallback” means

“Web fallback” means:

  • If the app is running as a website in a browser, it uses browser storage (localStorage) to save/load data.

  • If the app is running as a real mobile app (Android/iOS wrapper), it should use native phone storage instead.

So fallback = “default web mode when native mobile APIs are not available”.

Why I mentioned it for your project

Right now your persistence is purely browser-based:
